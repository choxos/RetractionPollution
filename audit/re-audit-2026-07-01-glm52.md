# RetractionPollution Re-Audit After GLM 5.2 Changes

**Audit timestamp:** 2026-07-01 UTC / 2026-06-30 22:39 EDT  
**Scope:** Current checkout after GLM 5.2 changes, R package source, CLI wiring,
generated `outputs/opencitations/` artifacts, and
`legacy/data/processed/opencitations.duckdb`.  
**Mode:** Audit only. I did not change package code or generated study results
for this review. I did create this audit file.  
**Workflow:** Sequential fallback for the statistical-review roles. The
subagent tool is available, but tool policy requires explicit user permission
before spawning agents.

## Bottom Line

The current checkout is materially better than the first study audit: seed
canonicalization is now present, the inspected DB has duplicate DOI seed rows
marked as provenance rather than canonical seeds, provenance metadata are
present, reports exist, and the R unit tests pass.

It is still not publication-ready or release-ready. The biggest unresolved study
blocker is unchanged: the existing OpenCitations crawl is incomplete, with
677,884 unprocessed depth 0/1 parents, so the exported depth-2 graph is a lower
bound rather than a complete census. GLM's changes also left package/release
regressions: the built package has an invalid `LICENSE` pointer, the README run
commands call functions that do not exist, and the standalone validation CSV no
longer agrees with the report.

## What Is Now Fixed

- Duplicate DOI seed records no longer contaminate the canonical top-seed table:
  `top_polluted_seeds.csv` has 60,305 rows, 60,305 unique `openalex_id` values,
  and 0 duplicate canonical seed rows.
- The existing DB now has seed statuses of `resolved = 60,305`,
  `duplicate_doi = 2,032`, and `no_doi = 5,912`.
- Linked duplicate seed provenance is retained: 1,603 linked duplicate DOI
  groups and 3,635 linked duplicate rows remain traceable.
- Required provenance keys exist in `run_metadata`: `rw_snapshot_date`,
  `oc_access_date`, `openalex_access_date`, `pipeline_mode`,
  `opencitations_seed_stats`, `last_crawl_summary`, and `depth3_truncated`.
- `outputs/opencitations/report.md` and `outputs/opencitations/report.html`
  exist and disclose the incomplete-depth validation failures.
- `R CMD build .` succeeds and the built package excludes the large `legacy/`,
  `outputs/`, and audit folders.

## Major Findings

### 1. The existing crawl is still incomplete, so complete-depth claims remain invalid

Severity: Major for the study.

Direct DB query of `frontier_nodes`:

| depth | nodes | processed | pending |
|---:|---:|---:|---:|
| 0 | 60,305 | 60,299 | 6 |
| 1 | 773,146 | 95,268 | 677,878 |
| 2 | 1,822,090 | 0 | 1,822,090 |

The validation table correctly fails
`frontier_processed_for_depth2_claim` with `observed = 677884` and
`expected = 0`. This means the depth-2 graph was built from only a subset of
depth-1 parents. The headline counts are lower bounds unless the crawl is
completed.

Required modification: rerun the OpenCitations crawl until every depth 0 and
depth 1 parent is processed, or relabel all depth-2 metrics as incomplete,
partial, or lower-bound counts.

### 2. `artifact_validation.csv` disagrees with the report

Severity: Major for reproducibility.

The standalone validation artifact currently says:

```text
report_exists,major,not_checked,TRUE,TRUE,...
```

But `outputs/opencitations/report.md` says:

```text
report_exists | major | pass | TRUE | TRUE | ...
```

The file timestamps also differ: `report.md` and `report.html` are from
21:46 EDT, while `artifact_validation.csv` is from 22:03 EDT. The report and the
machine-readable validation table are no longer the same validation pass.

Required modification: make the final report-generation step always persist the
same validation table that is embedded in the report, then regenerate the
`outputs/opencitations/` report and validation artifacts from current code.

### 3. The built R package has an invalid MIT license pointer

Severity: Major for release.

`DESCRIPTION` declares:

```text
License: MIT + file LICENSE
```

But `.Rbuildignore` excludes `LICENSE`, so the built tarball contains no
`retractionpollution/LICENSE`. `R CMD check --no-manual --no-build-vignettes
retractionpollution_0.1.0.tar.gz` reports:

```text
WARNING
Invalid license file pointers: LICENSE
```

The older re-audit claim that `LICENSE` is a two-line stub is stale: the live
checkout has the full MIT text again. The current problem is now package
metadata consistency. For `MIT + file LICENSE`, R expects a valid package
license file to be included, usually the two-field DCF stub.

Required modification: either include the proper R MIT `LICENSE` stub in the
tarball, or change the license declaration/file layout consistently.

### 4. The README run commands are not callable

Severity: Major for usability.

`README.md` tells users to run:

```bash
Rscript -e 'retractionpollution::run_all()'
Rscript -e 'retractionpollution::run_opencitations()'
```

After installing the built tarball into a temporary library, both symbols are
absent:

```text
run_all=FALSE
run_opencitations=FALSE
cmd_run_all=TRUE
```

The graph and CLI code show `cmd_run_all()` and `cmd_run_opencitations()` plus
`main()`, not public wrapper functions named `run_all()` or
`run_opencitations()`.

Required modification: add real exported wrapper functions matching the README,
or rewrite the README to use the actual CLI entrypoint.

### 5. Hybrid `cmd_run_all()` does not apply duplicate-status backfill

Severity: Major for hybrid runs; not a current OpenCitations-output failure.

The standalone hybrid preparation path calls `mark_duplicate_seed_resolutions()`
in `cmd_prepare_seeds()` and `cmd_resolve_seeds()`. The OpenCitations-only
integrated path also calls it in `cmd_run_opencitations()`.

The integrated hybrid path `cmd_run_all()` does not call
`mark_duplicate_seed_resolutions()` after `resolve_pending_seeds()`. A fresh
hybrid all-in-one run can therefore leave multiple resolved records for the same
`openalex_id`, causing `duplicate_canonical_seed_records` to fail and making
summary duplicate counts depend on which CLI path was used.

Required modification: call `mark_duplicate_seed_resolutions(store)` in
`cmd_run_all()` after seed resolution and before crawling/analysis.

## Remaining Study Limitations

### Stored and exported edge counts still differ

Validation still fails `stored_vs_exported_edges`:

```text
stored=3529449; exported_like=3522168; missing_targets=7281
```

This is now disclosed in the report as an induced-frontier export, which is an
improvement. It remains a real interpretation limit: the exported depth-2 CSV is
not the full stored edge table.

### OpenCitations failed parents remain unresolved

The DB has 52 `opencitations_failed_parent:*` metadata entries, and validation
fails `opencitations_failed_parents`. The report discloses this, but the failed
parents should be classified before publication, for example invalid DOI syntax
versus transient API failure.

### Topic and bridge-title outputs remain uninterpretable in OpenCitations-only mode

Validation still fails `topic_labels_present` and `bridge_titles_present`.
That is expected in the OpenCitations-only output because the crawler stores DOI
network structure without OpenAlex metadata enrichment. Topic distribution and
human-readable bridge-paper interpretation should remain suppressed or clearly
labeled unavailable for this run mode.

### The post-notice metric is still descriptive only

`summary.csv` reports `post_notice_direct_citers = 317,374`, but there is no
time-at-risk model, no pre/post rate denominator, no uncertainty interval, and
no matched non-retracted comparator. The report should avoid implying an effect
of notices on citation behavior.

Results:

- Unit tests passed, with 1 CRAN-only skip.
- Unit-test warnings were limited to local `duckdb` and `igraph` packages being
  built under R 4.6.1.
- `R CMD build .` passed.
- Built-tarball `R CMD check` exited 0 with 3 warnings:
  - local `igraph`/`duckdb` build-version warnings;
  - invalid `LICENSE` pointer;
  - undocumented exported objects.

The documentation warning is caused by broad `exportPattern("^[[:alpha:]]+")`
combined with no generated Rd files for the many user-level objects. This is
not a study-validity problem, but it is release debt.

## Final Verdict

Do not treat the current outputs as publication-ready. The codebase has moved in
the right direction, especially by making duplicate DOI rows explicit and by
adding validation, but the study still has a major incomplete-crawl blocker and
the release surface has concrete breakage. The next modification round should
prioritize: complete or relabel the crawl, regenerate consistent validation and
report artifacts, fix the package license metadata, fix the README API, and add
duplicate-status backfill to the hybrid `cmd_run_all()` path.
