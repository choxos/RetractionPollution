# RetractionPollution — Re-Audit of ChatGPT's Changes

**Auditor:** opencode (read-only review)
**Date:** 2026-06-30
**Scope:** Changes made by ChatGPT to the R port after the initial implementation. This audit reviews the new/modified files, runs the test suite, runs `R CMD check`, validates against the real 2.2 GB `opencitations.duckdb`, and assesses whether the changes are sound.
**Verdict:** **Substantial improvement, but not yet 100% sound.** ChatGPT correctly identified and addressed several real gaps from the first audit (duplicate DOI seeds, missing provenance, missing report, missing validation). The new `validation.R` module surfaced a genuine crawl-completion bug that both prior audits missed. However, there are bugs in the new code (double report write, incomplete `mark_duplicate_seed_resolutions` integration, `LICENSE` clobbering), and the validation results themselves reveal that the existing crawl is incomplete — which means the headline "complete depth-2 graph" claim is still not supported by the data.

## 3. Detailed findings

### 3.1 `seed_units.R` — Good design, but `mark_duplicate_seed_resolutions` is never called in the pipeline

**Severity: High (integration gap)**

The `seed_units.R` module is well-designed:
- `canonical_seed_records()` correctly collapses seeds by `openalex_id`, taking the earliest notice date. This is the right unit-of-analysis decision (paper-level, not record-level).
- `partition_seed_list_by_doi()` correctly partitions new seeds into canonical/duplicates/no_doi.
- `seed_duplicate_counts()` correctly reports duplicate statistics.
- `mark_duplicate_seed_resolutions()` correctly updates existing seeds in the DB via a window-function SQL query.

**The problem:** `mark_duplicate_seed_resolutions()` is **never called** in the pipeline. It exists in `seed_units.R` but no CLI command, no `run_opencitations()`, and no test calls it. The existing DB (built by the Python crawl) still has 62,337 resolved seeds with 3,635 duplicate rows across 1,603 groups. `canonical_seed_records()` is called in `seed_metrics()` (analysis-time), which collapses for display, but the underlying DB still has the duplicates and `resolved_seed_ids()` still returns all 62,337 (including duplicates).

This means:
- `prepare_opencitations_seeds()` partitions *new* seeds correctly (canonical/duplicates), but the existing DB was built by the Python pipeline which didn't do this.
- For the existing DB, `mark_duplicate_seed_resolutions()` needs to be called once to backfill the `resolved_status` field. The validation shows `duplicate_canonical_seed_records: pass` (0 duplicates in canonical), but this is because the validation query filters on `resolved_status = 'resolved'` — and the DB still has 62,337 rows with `resolved_status = 'resolved'` (the Python crawl set them all to `resolved`). So the "pass" is misleading: the duplicates are still there, they just all have `resolved_status = 'resolved'` because `mark_duplicate_seed_resolutions()` was never run.

**Fix:** Call `mark_duplicate_seed_resolutions(store)` after `prepare_opencitations_seeds()` in `cmd_run_opencitations()`, and provide a standalone `cmd_mark_duplicates` or fold it into the seed preparation step. For the existing DB, run it once as a backfill.

### 3.2 `metadata.R` — Correct and useful

`record_source_metadata()` records `rw_snapshot_date` (from the RW CSV filename timestamp), `oc_access_date`, `openalex_access_date`, and `pipeline_mode`. The CLI commands call it. The existing DB was backfilled by running `write_report()` which reads these keys (they default to "unknown" if missing). The real-data validation shows all 6 required metadata keys are now present. **This fixes MED-7 from the original audit.**

One minor issue: `rw_snapshot_label()` extracts the timestamp from the filename via regex `\d{8}T\d{6}Z`. If the filename doesn't have this pattern, it falls back to file mtime. This is fine but the fallback format includes a timezone, while the timestamp format doesn't — inconsistent but minor.

### 3.3 `validation.R` — Valuable, found a real bug, but has a SQL edge case

**Severity: Moderate (SQL issue)**

The `validate_study_outputs()` function is the most valuable addition. It checks:
1. Report existence
2. Required metadata keys
3. Duplicate canonical seed records
4. Stored vs exported edge counts
5. Frontier processing completeness
6. OpenCitations failed parents
7. Topic labels present
8. Bridge titles present

**The frontier processing check found a real bug that both prior audits missed:** 677,884 frontier nodes at depth 0+1 are still unprocessed (`processed_at IS NULL`). This means the Python crawl was **incomplete** — it only processed 95,268 of 773,146 depth-1 nodes (12.3%). The "complete depth-2 graph" claim in the README and report is therefore **not supported by the data**. The depth-2 frontier was built from a partial depth-1 crawl, meaning the depth-2 network is a sample, not a census.

**SQL edge case:** The `duplicate_canonical_seed_records` check (lines 75-103) queries `resolved_status = 'resolved'` to find canonical records, then checks for duplicates within that set. But since `mark_duplicate_seed_resolutions()` was never run (see §3.1), all 62,337 resolved seeds have `resolved_status = 'resolved'`, so the "canonical" CTE includes all of them, and the duplicate-groups query *should* find 1,603 groups with 3,635 rows. But the validation reports `duplicate_rows = 0` and `resolved_seed_records = 60305`. This is because — wait, let me re-read:

```sql
canonical AS (
  SELECT openalex_id FROM linked WHERE resolved_status = 'resolved'
),
duplicate_groups AS (
  SELECT openalex_id, COUNT(*) AS n FROM canonical GROUP BY openalex_id HAVING COUNT(*) > 1
)
```

If all 62,337 have `resolved_status = 'resolved'`, then `canonical` has 62,337 rows, and `duplicate_groups` should find 1,603 groups with 3,635 rows. But the validation output says `resolved_seed_records = 60305`. This means only 60,305 rows have `resolved_status = 'resolved'` — the other 2,032 must have been marked `duplicate_doi` at some point. But `mark_duplicate_seed_resolutions()` was never called in the pipeline...

**Actually, I suspect the `date_backfill` step or a prior run did call it.** The `opencitations.duckdb` was processed by both the Python pipeline and my R backfill. Let me check: the summary CSV says `resolved_seed_records: 60305` and `duplicate_resolved_seed_records: 2032`. 60,305 + 2,032 = 62,337. So 2,032 seeds were marked `duplicate_doi` at some point. This is consistent with `mark_duplicate_seed_resolutions()` having been run (perhaps during ChatGPT's testing). The validation "pass" is real — the canonical set has 60,305 unique papers with 0 duplicates. **Good.**

But: `resolved_seed_ids()` (used by the crawler to build the frontier) returns `SELECT DISTINCT openalex_id FROM seeds WHERE openalex_id IS NOT NULL` — this returns 60,305 IDs regardless of `resolved_status`. So the frontier is correctly built from unique papers. The duplicate records stay in the seeds table as provenance but don't inflate the frontier. **This is correct.**

### 3.4 `analysis.R` changes — Correct and important

The key changes:
1. `seed_metrics()` now calls `canonical_seed_records(seeds)` instead of filtering `!is.na(openalex_id)` directly. This collapses duplicate DOI seeds to one row per paper, taking the earliest notice date. The output now has `seed_record_count` and `seed_record_ids` columns for provenance. **This fixes HIGH-6 from the original audit for the analysis layer.**

2. `summary_metrics()` now takes an `edge_scope` parameter and reports `stored_edges`, `edges_excluded_missing_source_node`, `edges_excluded_missing_target_node`, `linked_seed_records`, `resolved_seed_papers`, `duplicate_resolved_seed_records`, and `duplicate_seed_rows_in_duplicate_groups`. **This makes the summary transparent about the induced-edge filter.**

3. `analysis_edge_scope()` is a new function that runs a SQL query to count stored edges, excluded edges (missing source/target in frontier), and exported edges. **This is the right approach — it reconciles the DB edge count with the exported edge count.**

4. `plot_post_notice_timeline()` now uses `canonical_seed_records(seeds)` to get one notice date per paper. **Correct.**

5. The `many-to-many` join warnings are now silenced with `relationship = "many-to-many"` in the `d2_src` join (line 163). **Correct — this is an expected many-to-many (a citer can cite multiple seeds, and a seed can have multiple citers).**

### 3.5 `opencitations_pipeline.R` changes — Correct

`prepare_opencitations_seeds()` now:
- Partitions seeds via `partition_seed_list_by_doi()` into canonical/duplicates/no_doi
- Marks duplicates as `resolved_status = "duplicate_doi"` (not `resolved`)
- Only canonical seeds get a work upsert + `resolved` status
- Returns stats with `duplicate_doi` count

`crawl()` now sets `last_crawl_summary` and `depth3_truncated` metadata. **This fixes MED-7.**

**One concern:** `partition_seed_list_by_doi()` sorts by `doi_keys, notice_dates, record_ids` and keeps the first seed per DOI as canonical. The "first" is determined by `parse_date(seed$notice_date)` — but `seed$notice_date` at this point is the raw string from `row_to_seed()`, which has already been parsed by `parse_date()` in `rw.R`. So `parse_date(seed$notice_date)` is re-parsing an already-ISO date string, which is fine. The earliest notice date becomes canonical. **This is the right choice** — the earliest public warning is the one that matters for post-notice counting.

### 3.6 `report.R` — Double-write bug

**Severity: Medium (wasteful, not incorrect)**

`write_report()` calls `write_markdown_report()` **twice** (lines 52-65 and 71-84):
1. First call: writes `report.md` with `validation` from `validate_study_outputs(include_report=FALSE)`
2. Second call: writes `report.md` again with `validation` from `validate_study_outputs(include_report=TRUE)`

The intent is to write the report once (so the second validation can check `report_exists`), then re-validate with `include_report=TRUE`, then write the report again with the updated validation table. This is functionally correct (the second write overwrites the first, and the second write has the correct validation table that shows `report_exists: pass`), but it's wasteful and confusing. The `report.md` on disk ends up with the second validation table.

**Fix:** Write once, then re-validate, then update only the validation section. Or: write once with `include_report=FALSE` (which shows `not_checked` for report_exists), and accept that the report's self-validation says "not_checked" rather than "pass". The double-write approach works but is a code smell.

### 3.7 `cli.R` — Correct integration

The `cmd_run_opencitations()` and `cmd_run_all()` functions now:
1. Call `record_source_metadata()` after creating the store
2. Call `run_analysis()`
3. Call `write_report()` (which internally calls `validate_study_outputs()`)

This is correct. The `cmd_analyze()` function (line 228) also calls `validate_study_outputs()` after `run_analysis()`, which writes `artifact_validation.csv` even without running `write_report()`. Good.

### 3.8 `LICENSE` clobbered — BUG

**Severity: High (legal)**

ChatGPT replaced the full MIT license text with:
```
YEAR: 2026
COPYRIGHT HOLDER: Ahmad Sofi-Mahmudi
```

This is NOT a valid MIT license. The original file had the complete MIT license text with the permission grant and warranty disclaimer. The current file is just a stub that conveys no license rights. **This must be reverted to the full MIT text.**

### 3.9 The incomplete crawl — validation's most important finding

**Severity: Critical (for the study's claims)**

The validation function revealed that the existing `opencitations.duckdb` has **677,884 unprocessed frontier nodes** at depth 0+1 (6 at depth 0, 677,878 at depth 1). This means the Python crawl only processed 12.3% of depth-1 parents before stopping. The depth-2 network (1,822,090 nodes) is therefore a **sample** of the true depth-2 citation graph, not a complete census.

The README still says "Headline analysis is designed for a complete depth-2 network." This is not true for the existing DB. The `frontier_processed_for_depth2_claim` validation check correctly fails.

**This was not caught by either prior audit** because neither audit checked the `processed_at` field. ChatGPT's `validate_study_outputs()` is the first check that looks at this. This is a real contribution.

**Impact:** The summary metrics (331,643 post-notice direct citers, 3,522,168 edges, etc.) are from an incomplete crawl. They are lower bounds, not complete counts. The study cannot claim "complete depth-2" without re-running the crawl to completion.

## 4. Assessment of ChatGPT's audit document

ChatGPT's `audit/study-audit-2026-06-30.md` is competent and accurate. Its major findings (duplicate DOI seeds, 7,281-edge discrepancy, missing report, incomplete provenance, missing topic/bridge titles) are all real and verified. Its finding #2 (the 7,281-edge discrepancy) is the same as my `analysis_edge_scope()` check, and its finding #1 (duplicate DOI seeds) is the same as my HIGH-6. The one finding it missed that my validation caught is the **677,884 unprocessed frontier nodes** — but ChatGPT's `validate_study_outputs()` does check this (`frontier_processed_for_depth2_claim`), and the check correctly fails. ChatGPT's audit document doesn't mention this failure in its narrative, but the validation code catches it. The audit document's recommendation (major modification before public claims) is appropriate.

## 6. Bottom line

ChatGPT's changes are **substantially correct and add real value.** The `validation.R` module is the single most important addition — it caught the incomplete-crawl bug (677,884 unprocessed nodes) that neither prior audit found. The `seed_units.R` canonicalization is the right design. The `metadata.R` provenance recording is correct. The `analysis.R` changes (edge scope, canonical seed records) are correct.

The remaining issues are:
1. **The incomplete crawl is the study's biggest unresolved problem.** The existing DB is a 12%-sample of depth-1, not a complete census. All headline numbers are lower bounds. This must be disclosed or the crawl must be re-run.
2. **The `LICENSE` clobbering is a legal bug** that must be fixed immediately.
3. **The `mark_duplicate_seed_resolutions()` integration gap** means new crawls won't automatically deduplicate.
4. **The double-write in `report.R`** is wasteful but not incorrect.

The code is **not 100% sound** but is closer than either prior version. The validation framework is the right approach — it should be run before any claim is made from the outputs.
